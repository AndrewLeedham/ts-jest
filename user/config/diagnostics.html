Diagnostics option

The diagnostics option allows to enable/disable the error reporting. But it can also be used to filter which one should throw during tests, as well from which file it should be reported or not.

A diagnostic can be:

  • an error in your TypeScript config (whether it‚Äôs a file or inline)
  • syntax errors in some of your TypeScript files (source or tests)
  • type/semantic errors, what TypeScript has actually been made for üòÅ

If a diagnostic is not filtered out, it‚Äôll fail the compilatin within TSJest, and so will your related test.

Disabling/enabling

By default all diagnostic are enabled. This is the same as setting the diagnostics option to true. To disable all diagnostics, set diagnostics to false (you might experience slightly better performence as well, especially if you disabled Jest cache).

Advanced configuration

The option‚Äôs value can also accpet an object for more advanced configuration. Each config. key is optional:

  • pretty: Enables/disable colorful and pretty output of errors (default: enabled).
  • ignoreCodes: List of TypeScript error codes to ignore. Complete list can be found there. By default here are the ones ignored:
    • 6059: ‚ÄòrootDir‚Äô is expected to contain all source files.
    • 18002: The ‚Äòfiles‚Äô list in config file is empty. (it is strongly recommanded to include this one)
    • 18003: No inputs were found in config file.
  • pathRegex: If specified, diagnostics of source files which path does not match will be ignored.

Examples

Disabling diagnostics:

// jest.config.js
module.exports = {
  // [...]
  globals: {
    'ts-jest': {
      diagnostics: false
    }
  }
};

// OR package.json
{
  // [...]
  "jest": {
    "globals": {
      "ts-jest": {
        "diagnostics": false
      }
    }
  }
}

Advanced options:

Enabling diagnostics for test files only

Assuming all your test files ends with .spec.ts or .test.ts, using the following config will enable error reporting only for those files:

// jest.config.js
module.exports = {
  // [...]
  globals: {
    'ts-jest': {
      diagnostics: {
        pathRegex: /\.(spec|test).ts$/
      }
    }
  }
};

// OR package.json
{
  // [...]
  "jest": {
    "globals": {
      "ts-jest": {
        "diagnostics": {
          "pathRegex": "\\.(spec|test)\\.ts$"
        }
      }
    }
  }
}
Ignoring some error codes:

All TypeScript error codes can be found there. The ignoreCodes option accepts this values:

  1. A single number (example: 1009): unique error code to ignore
  2. A string with a code (example "1009", "TS1009" or "ts1009")
  3. A string with a list of the above (example: "1009, TS2571, 4072 ,ts6031 ")
  4. An array of one or more from 1 or 3 (example: [1009, "TS2571", "6031"])

It‚Äôs advised to use concise types along the list of course:

// jest.config.js
module.exports = {
  // [...]
  globals: {
    'ts-jest': {
      diagnostics: {
        ignoreCodes: [2571, 6031, 18003]
      }
    }
  }
};

// OR package.json
{
  // [...]
  "jest": {
    "globals": {
      "ts-jest": {
        "diagnostics": {
          "ignoreCodes": [2571, 6031, 18003]
        }
      }
    }
  }
}
